### Machine Learning: Steam Game Success Prediction

이 프로젝트는 스팀 게임 데이터를 기반으로 특정 게임이 흥행(Hit)할지 여부를 예측하는 분류 모델을 구축하는 데 목적이 있습니다. 83개의 정형 피처를 활용하여 비즈니스 가치가 높은 흥행작 탐지 성능을 개선하는 과정에 집중했습니다.

---

#### 1. 문제 정의 및 데이터 불균형 해결 (Baseline: Random Forest)

초기 모델 구축 단계에서 랜덤 포레스트를 활용했으나, 데이터 불균형 문제로 인해 모델이 비흥행작 위주로 학습되는 편향이 발생했습니다.

* **문제 상황**: 전체 데이터 중 성공작(1)의 비율이 약 18%에 불과하여, 모델이 대다수인 실패작(0)으로 예측하는 경향이 매우 강했습니다.
* **해결 전략**: **데이터 레벨 다운샘플링(Downsampling)** 적용
    * 비흥행작(0) 데이터를 성공작(1) 데이터 수와 동일하게(1:1) 맞추는 무작위 비복원 추출을 수행했습니다.
    * **이유**: 단순 정확도(Accuracy)의 착시를 제거하고, 플랫폼 기획의 핵심인 '흥행작 탐지 성능(Recall)'을 정량적으로 확보하기 위함입니다.

---

#### 2. 모델 고도화: 알고리즘 교체 (LightGBM)

기본 모델에서 확보한 데이터 균형을 바탕으로, 더 정교한 패턴 학습을 위해 LightGBM 알고리즘으로 모델을 교체했습니다.

* **모델 선정 이유**: 
    * **패턴 학습 능력**: 랜덤 포레스트보다 복잡한 데이터 관계를 학습하는 데 효과적인 부스팅 알고리즘을 도입하여 정밀도와 재현율을 동시에 개선하고자 했습니다.
    * **효율성**: 데이터가 증가하더라도 연산 자원을 효율적으로 사용하여 빠른 모델 학습 및 배포가 가능합니다.
* **실험 결과**: 
    * 재현율(Recall) 0.80, 정밀도(Precision) 0.46을 확보하여 이전 모델 대비 전반적인 예측 균형(F1-score 0.58)을 높였습니다.

---

#### 3. 딥러닝 기반 검증 및 모델 개선 (PyTorch MLP)

동일한 1:1 균형 정형 데이터셋 환경에서 트리 기반 모델 외에 **심층 신경망(MLP)**이 복잡한 비선형 패턴을 학습할 수 있는지 추가 검증을 진행했습니다.

* **초기 문제점 분석**:

  * 초기 MLP 구현에서는 학습 과정에서 Test Set 성능을 직접 모니터링하며 최적 모델을 선택하는 구조를 사용했습니다.
  * 이 방식은 테스트 데이터가 모델 선택 과정에 간접적으로 활용되는 문제가 있어 실제 일반화 성능을 과대평가할 가능성이 존재했습니다.
  * 또한 딥러닝 모델 학습에 별도의 입력 데이터 스케일링이 적용되지 않아 정형 데이터 환경에서 학습 안정성이 떨어질 수 있었습니다.

* **개선 사항**:

  * 학습 데이터 내부에서 **Validation Set을 별도로 분리**하여 모델 선택에 사용하고, Test Set은 최종 평가 단계에서만 활용하도록 학습 구조를 수정했습니다.
  * **StandardScaler**를 적용하여 모든 입력 피처를 정규화함으로써 학습 안정성과 수렴 성능을 개선했습니다.
  * 과적합을 방지하기 위해 **Dropout 비율을 0.3 / 0.2에서 0.4 / 0.3으로 상향 조정**했습니다.
  * Validation Loss 기준으로 가장 우수한 성능을 기록한 시점의 가중치를 자동 저장하는 **Best Model 저장 로직**을 적용했습니다.

* **실험 결과**:

  * 기존 구현에서는 Test Set을 기준으로 최적 가중치를 선택하여 Accuracy 86%, Precision 0.54, Recall 0.75, F1-Score 0.62를 기록했습니다.
  * 그러나 이는 모델 선택 과정에서 Test Set 정보가 간접적으로 활용된 구조였기 때문에 실제 일반화 성능보다 다소 높게 측정되었을 가능성이 있었습니다.
  * 이를 개선하기 위해 Validation Set 기반 학습 구조와 StandardScaler를 적용한 결과, 최종 Test Set 기준 Accuracy 83.10%, Precision 0.48, Recall 0.82, F1-Score 0.61을 기록했습니다.
  * 정확도와 정밀도는 소폭 감소했지만, 프로젝트의 핵심 목표인 흥행작 탐지 성능(Recall)은 0.75 → 0.82로 향상되었습니다.
  * 반면 F1-Score는 0.62 → 0.61으로 소폭 감소했으나, Validation 기반 평가 체계를 적용했음에도 유사한 수준의 성능을 유지했습니다.
  * 결과적으로 기존 모델 대비 과적합 가능성을 줄이면서도 흥행작 탐지 능력을 향상시킨 보다 신뢰도 높은 평가 체계를 구축할 수 있었습니다.

#### 4. 모델 성능 요약

실험을 통해 도출된 알고리즘별 최종 테스트 결과입니다. 모든 모델은 데이터 불균형 문제를 해소하기 위해 동일한 다운샘플링 데이터셋을 기반으로 학습되었습니다.

**1. 랜덤 포레스트: 초기 상태 (데이터 불균형)**

데이터 불균형으로 인해 모델이 대다수인 '0(비흥행작)'에 치중하여 학습되었을 때의 결과입니다. (흥행작 탐지 성능인 Recall이 현저히 낮음)

#### 5. 웹 시뮬레이터 배포 및 서빙 과정의 트러블슈팅

학습 완료된 모델의 비즈니스 효용성을 검증하기 위해 **Streamlit 기반의 '게임 스펙 추론 시뮬레이터' 대시보드**를 구현했습니다.

* **생산 환경 최적화**: 신경망 말단의 `nn.Sigmoid()`를 제거하고 raw 출력인 로짓(Logit)만 서빙 엔진이 연산하도록 아키텍처를 경량화했습니다. 이를 통해 모델의 범용성을 높이고 PyTorch 표준 규격에 맞춘 수치 안정성을 확보했습니다.
* **UX 관점의 데이터 파이프라인**: 복잡한 83개의 인코딩 피처 입력을 UI단에서 '게임플레이'와 '비주얼/기획 테마'의 2단계 멀티셀렉트 카테고리로 분리하여 입력 편의성을 극대화하고, 백엔드 파이프라인에서 이를 다시 단일 텐서로 병합·추론하는 정석적인 서빙 아키텍처를 구축했습니다.
* **프로덕션 환경의 실무적 타협 (데이터 불일치 문제 대응)**: 
    * 원래 기획은 하나의 대시보드 안에서 머신러닝(LightGBM) 결과와 딥러닝(MLP) 결과를 동시에 보여주며 비교하는 구조였습니다.
    * 하지만 딥러닝 모델을 업그레이드하는 과정에서 '텍스트 데이터(설명글)'를 추가로 활용하게 되었고, 이로 인해 머신러닝 모델이 사용하는 데이터 형태(83개 컬럼)와 딥러닝 모델이 요구하는 데이터 형태가 서로 달라지는 문제가 발생했습니다.
    * 제한된 시간 내에 대시보드의 안정성을 확보하기 위해, 우선은 입력 구조가 명확한 **딥러닝 전용 예측 시뮬레이터로 단일화하여 먼저 배포**하는 타협점을 택했습니다.

---

#### 향후 계획

* **비즈니스 목적에 따른 임계값(Threshold) 조정**: 현재 모델은 성공 확률 50%를 의미하는 기본 로짓 값 0.0을 기준으로 흥행 여부를 예측하고 있습니다. '흥행작 후보'를 하나도 놓치지 않는 것이 비즈니스적으로 더 중요하다면, 이 기준점(Threshold)을 0.0보다 더 낮추어 흥행작 탐지율(Recall)을 극대화하는 최적화 실험을 추가할 예정입니다.
* **모델 해석력 보완**: 어떤 게임 피처(Feature)가 흥행에 가장 큰 영향을 미쳤는지 모델의 변수 중요도(Feature Importance)를 시각화하여, 단순 예측을 넘어 흥행 요인을 분석하는 인사이트 도출 단계로 확장하고자 합니다.
* **대시보드 기능 복원 (ML과 DL 모델 비교 기능)**: 딥러닝 학습에 사용했던 텍스트 변환기(TF-IDF Vectorizer)를 파일 형태로 저장하여 웹 서비스단에 이식할 예정입니다. 이를 통해 사용자가 대시보드에 글을 입력하면 실시간으로 데이터를 변환하여, 머신러닝과 딥러닝 두 모델의 예측 결과를 한 화면에서 바로 비교할 수 있도록 데이터 파이프라인을 통합할 계획입니다.
