# 자연어 게임 검색 — ML vs DL 의미 검색 비교

> 이 문서는 의미 검색 기능의 설계·구현·결과를 발표용으로 정리한 것입니다.
> 숨은 명작 탐지(정형 데이터)와 대비되는 두 번째 ML vs DL 비교 사례입니다.

---

## 1. 문제 정의 — 스팀의 실제 불편

스팀에는 게임이 9만 개 이상 있지만, 유저가 원하는 게임을 찾는 방법은
태그 필터 조합뿐이다. "잔잔한 힐링 게임"을 찾으려면 Relaxing, Casual,
Singleplayer 태그를 직접 알고 조합해야 한다.

자연어("잔잔한 힐링 게임")를 입력하면 의미에 맞는 게임을 찾아주는
검색 시스템이 있다면 이 불편을 해결할 수 있다.

이 검색을 **ML과 DL 두 가지 방식으로 구현하고 성능을 비교**하는 것이
이 기능의 목적이다.

---

## 2. 왜 이 문제에서 ML vs DL 비교가 의미 있는가

**숨은 명작 탐지**(정형 데이터)에서는 ML(XGBoost)이 DL(MLP)을 이겼다.
피처가 숫자 83개인 표 형태에서는 트리 기반 부스팅이 신경망보다 효율적이었다.

그런데 **의미 검색**은 입력이 텍스트다. 텍스트는 단어 순서, 문맥, 동의어,
다국어 같은 복잡한 구조가 있어서, 단순한 단어 빈도(ML)로는 포착이 어렵고
의미 벡터(DL)가 유리할 수 있다.

한 프로젝트 안에서 **"정형 데이터에선 ML이 이기고, 텍스트에선 DL이 이기는"
대조**를 보여주면, "무엇이 더 좋은가"가 아니라 **"데이터 성격이 모델 선택을
결정한다"**는 핵심 결론에 도달한다.

---

## 3. ML 방식 — TF-IDF 단어 매칭

### 원리
게임 설명문을 **단어 빈도(TF-IDF)** 벡터로 변환하고,
유저 쿼리도 같은 방식으로 벡터화해서 코사인 유사도로 비교한다.

```
"A relaxing cozy farming sim..." → [relaxing: 0.32, cozy: 0.28, farming: 0.41, ...]
"relaxing game"                 → [relaxing: 0.71, game: 0.29, ...]
→ 코사인 유사도 = 0.51 (높을수록 관련)
```

### 강점
- 빠르다 (벡터 계산이 단순 행렬곱)
- 단어가 정확히 겹치면 매우 정확 (horror → Horror 태그 90%)
- 모델 다운로드가 필요 없음

### 약점 (핵심 한계)
- **단어가 겹쳐야만 매칭된다.** "힐링"과 "relaxing"이 같은 뜻임을 모름
- 한국어 쿼리로 영어 설명문 검색 불가
- "challenging game I can die and retry"처럼 의역하면 매칭 실패

### 구현
scikit-learn의 `TfidfVectorizer` (max_features=5000, ngram=(1,2))

---

## 4. DL 방식 — Sentence-Transformer 의미 임베딩

### 원리
문장 전체를 **의미 벡터**(384차원)로 변환하는 딥러닝 모델을 사용한다.
"잔잔한 힐링 게임"과 "A relaxing cozy game"이 의미가 비슷하면
벡터가 가까운 곳에 놓인다. 단어가 하나도 겹치지 않아도 매칭 가능.

```
"잔잔한 힐링 게임"           → [0.12, -0.34, 0.56, ...] (384차원)
"A relaxing cozy farming..." → [0.14, -0.31, 0.52, ...] (384차원)
→ 코사인 유사도 = 0.82 (의미적으로 가까움)
```

### 강점
- **의미로 매칭한다.** 동의어, 의역, 비유 표현도 잡음
- **다국어 지원.** 한국어 쿼리 → 영어 설명문 교차 검색 가능
  (50개 언어를 같은 의미 공간에 매핑)
- "시간 순삭되는 게임"처럼 설명문에 없는 표현도 의미로 찾음

### 약점
- 임베딩 모델 다운로드 필요 (~120MB, 최초 1회)
- 단순 키워드 검색에선 TF-IDF보다 느림
- 의미를 "해석"하기 때문에 가끔 직관과 다른 결과

### 구현
`sentence-transformers` 라이브러리의
`paraphrase-multilingual-MiniLM-L12-v2` 모델.
50개 언어를 공통 의미 공간에 매핑해 다국어 교차 검색을 지원한다.

### 왜 한국어 전용 모델(KR-SBERT)이 아닌가
게임 설명문이 **전부 영어**라서, 한국어 전용 모델은 영어 문서를
제대로 임베딩하지 못한다. 한국어 쿼리와 영어 문서를 같은 공간에
놓으려면 **다국어 모델**이 필요하다.

---

## 5. 평가 방법 — 태그 기반 프록시 평가

검색은 "정답"이 없는 문제다. 그래서 게임에 이미 달려 있는
**태그를 정답으로 활용**했다:

```
쿼리 "relaxing game" → 결과 상위 20개 중 'Relaxing' 태그 게임 비율
= Precision@20
```

14개 쿼리를 세 유형으로 나누어 평가:

| 쿼리 유형 | 예시 | ML 유리/DL 유리 |
|---|---|---|
| 영어 키워드 (word) | "horror scary", "puzzle" | ML 유리 (단어 직접 매칭) |
| 자연어 문장 (natural) | "a calm game to unwind" | DL 유리 (의미 해석) |
| 한국어 (korean) | "잔잔한 힐링 게임" | DL만 가능 |

### 태그 그룹 정답 (시행착오에서 배운 점)
처음에 정답 태그를 "Roguelike"로 설정했더니 Precision이 0%가 나왔다.
원인: 데이터에 "Roguelike"라는 태그는 없고 "Rogue-like", "Rogue-lite",
"Action Roguelike"로 쪼개져 있었다. 검색은 멀쩡했고 **채점 기준이 틀린 것.**
→ 정답을 단일 태그가 아닌 **관련 태그 그룹**으로 수정해 해결.
(Roguelike 그룹: Rogue-like, Rogue-lite, Action Roguelike, Traditional Roguelike)

## 6. 실험 결과

### 쿼리 유형별 Precision@20

| 쿼리 유형 | ML (TF-IDF) | DL (임베딩) | 해석 |
|---|---|---|---|
| 영어 키워드 | **높음** | 대등 또는 약간 낮음 | 단어가 그대로 겹치니 ML이 강함 |
| 자연어 문장 | 중간 | **ML 대비 개선** | 의역·문맥을 DL이 잡음 |
| 한국어 | **0%** (매칭 불가) | **작동** | ML은 영어 설명문과 한국어 매칭 자체가 불가 |

### 핵심 발견

**① ML이 이기는 경우가 있다** — "horror", "puzzle" 같은 짧은 영어 키워드는
설명문에 그 단어가 대부분 직접 들어가 있어서 TF-IDF가 정확하고 빠르다.

**② DL만 할 수 있는 영역이 있다** — 한국어 쿼리("잔잔한 힐링 게임")는
ML로는 아예 불가능하다. 영어 설명문에 "잔잔한"이란 단어가 없으니까.
DL은 의미 공간에서 "잔잔한"≈"relaxing"을 알고 있어서 가능.

**③ "roguelike"가 둘 다 0%였던 이유** — 모델 문제가 아니라 평가 태그
이름이 틀렸다. 데이터에는 "Rogue-like"(하이픈)로 저장돼 있어서
"Roguelike"와 매칭이 안 됐음. 태그 그룹으로 수정 후 정상화(ML 75%).

## 7. 하이브리드 검색 — 둘의 강점을 합침

위 결과를 바탕으로, 쿼리 성격을 보고 **자동으로 최적 방식을 선택**하는
라우터를 구현했다:

```
"action"              → 짧은 영어 키워드  → ML (단어매칭이 빠르고 정확)
"horror scary"        → 짧은 영어 키워드  → ML
"잔잔한 힐링 게임"     → 한국어 포함      → DL (의미 임베딩만 가능)
"a relaxing game..."  → 자연어 문장      → DL (의미 해석 필요)
```

라우팅 규칙:
- 한국어가 포함되어 있으면 → DL (ML은 교차 매칭 불가)
- 3단어 이하 영어 → ML (단어 직접 매칭이 정확)
- 그 외 (자연어 문장) → DL (의미 해석)

유저는 어떤 방식이 선택됐는지 배지로 볼 수 있지만, **직접 선택할 필요는 없다.**

## 8. 결과 품질 개선 — 인기/평점 가중

순수 텍스트 유사도만으로 정렬하면 리뷰 20개짜리 무명 게임이 상위를 차지한다.
"관련성은 높지만 검증이 안 된" 게임. 실제 유저 입장에선 불만.

이를 해결하기 위해 **관련성 점수에 인기/평점 가중치를 결합**:

```
최종 점수 = (1-w) × 텍스트 관련성 + w × 인기 점수
인기 점수 = log(리뷰 수) × 긍정률, 0~1로 정규화
```

w=0이면 순수 관련성, w=0.35면 균형, w=0.6이면 인기 위주.
Streamlit에서 유저가 슬라이더로 조절 가능.

검증된 효과 (쿼리: "action"):
- 가중치 0: 리뷰 20~99개 무명 게임만 (Crypt Hunter, Toad's Soul Hopper)
- 가중치 0.4: Hogwarts Legacy(31만 리뷰), BlazBlue(긍정 95%) 등 검증된 게임

## 9. 전체 프로젝트 결론 — 두 사례의 대조

| | 숨은 명작 탐지 | 자연어 검색 |
|---|---|---|
| 데이터 유형 | 정형 (숫자 83개 피처) | 텍스트 (설명문) |
| ML 방식 | XGBoost (트리 부스팅) | TF-IDF (단어 빈도) |
| DL 방식 | MLP (신경망) | Sentence-Transformer (의미 임베딩) |
| 승자 | **ML 우세** (PR-AUC 0.279 vs 0.222) | **DL 우세** (한국어·자연어) |
| 이유 | 정형+중규모 데이터에서 트리 부스팅이 효율적 | 의미·다국어 매칭은 임베딩만 가능 |

**핵심 결론:**
> "어떤 모델이 더 좋은가"가 아니라 **"데이터의 성격이 모델 선택을 결정한다."**
> 정형 데이터에는 ML, 텍스트 의미 처리에는 DL. 이를 같은 프로젝트 안에서
> 두 사례로 실증했으며, 실제 서비스에서는 하이브리드로 결합하는 것이 최선이다.

## 10. 한계점

- 평가가 태그 기반 프록시 → 유저 만족도 직접 측정은 못 함
- 한국어 쿼리의 정량 평가가 어려움 (영어 설명문에 한국어 태그가 없어
  Precision 측정 불가, 정성 확인만 가능)
- 임베딩 모델이 게임 도메인에 특화되지 않음 (범용 다국어 모델)
- TF-IDF의 max_features(5000)와 임베딩 차원(384)이 공정 비교인지 논란 가능

## 11. 향후 개선

- 게임 도메인 특화 임베딩 모델 파인튜닝
- 유저 피드백(클릭/플레이) 기반 검색 품질 평가
- 게임 썸네일 이미지 임베딩 추가 (멀티모달 검색)
- 태그 자동 분류 모델로 태그 품질 사전 정제 후 검색
