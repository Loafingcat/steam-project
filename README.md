# Steam Games 전처리 파이프라인 — 설명서

> 데이터셋: Kaggle artermiloff/steam-games-dataset (`games_march2025_cleaned.csv`)
> 목적: ML vs DL 비교를 위한 흥행 예측 / 숨은 명작 탐지 데이터 생성

---

## 1. 한눈에 보기

| 항목 | 값 |
|---|---|
| 원본 | 89,618행 × 47열 (468MB) |
| 전처리 후 | 46,572행 × 83피처 |
| 타겟 | target_hit (흥행, 양성 15.6%) / target_hidden_gem (숨은 명작, 양성 19.7%) |
| 실행 | `uv run run_pipeline.py` (약 19초) |
| 산출물 | `data/processed/` — parquet 8개 + scaler 2개 + tfidf 3개 + meta JSON |

```
games_march2025_cleaned.csv (89,618행)
  │
  ├─[1] 로딩      대용량 13컬럼 제외 + 메모리 다운캐스팅
  ├─[2] 클리닝    미성숙작/저리뷰 제거 → 46,572행
  ├─[3] 라벨링    owners·긍정률로 타겟 2종 생성
  ├─[4] 피처      리스트 파싱·멀티핫·날짜 파생 → leakage 17컬럼 물리 제거
  └─[5] 분할/저장 stratified 8:2 → train에만 스케일링 → parquet
  │
  └─→ load_processed("hit" | "hidden_gem")  ← 팀원 인터페이스
```

## 2. 팀원 사용법 (모델 담당자는 이것만 알면 됨)

`src/pipeline.py`와 `data/processed/`를 같은 프로젝트에 두고:

```python
from pipeline import load_processed

# 흥행 예측
X_train, X_test, y_train, y_test = load_processed("hit")

# 숨은 명작 탐지
X_train, X_test, y_train, y_test = load_processed("hidden_gem")

# 텍스트(TF-IDF)까지 받기 — DL "정형 vs 정형+텍스트" 실험용
X_tr, X_te, y_tr, y_te, T_tr, T_te = load_processed("hit", with_text=True)
```

받는 데이터의 보장 사항:
- **스케일링 완료** (StandardScaler, train 기준)
- **leakage 차단 완료** — 출시 후 정보는 컬럼 자체가 없음. 아무 피처나 써도 안전
- **train/test 동일 분할** (random_state=42) — 팀원 간 비교 공정성 보장

## 3. 전처리 단계별 기준과 근거

### [1] 로딩
468MB 파일이라 메모리 관리가 첫 과제. `detailed_description`,
`screenshots`, `movies` 등 **대용량 텍스트·URL 13개 컬럼을 로딩 시점에 제외**하고,
숫자형은 float64→float32로 다운캐스팅했다.

### [2] 클리닝 — 4가지 제거 기준

| 기준 | 제거량 | 근거 |
|---|---|---|
| 출시 180일 미만 | 9,943건 | 라벨 신뢰성 (아래 상술) |
| 리뷰 10개 미만 | 33,103건 | 긍정률 표본 확보 |
| 출시일 파싱 실패 | 0건 | 날짜 포맷 정상 |
| 가격 $100 초과 클리핑 | 57건 | 이상값이 스케일링 왜곡 |

**▶ 왜 출시 180일 미만을 제거하나 (핵심 의사결정)**

두 라벨(owners, 긍정률) 모두 **출시 후 시간이 지나야 쌓이는 값**이다.
2025년 2월 출시작은 데이터 기준일(2025-03-10)까지 한 달뿐이라,
실제로는 흥행할 게임도 owners가 낮게 찍혀 "비흥행"으로 **잘못 라벨링**된다.
이건 모델이 극복할 노이즈가 아니라 **정답 자체가 틀린** 경우다.

180일(약 6개월)은 게임 초기 판매·리뷰가 안정화되는 통상 기간.
즉 이 데이터는 "출시 6개월 시점의 성과"를 기준으로 학습한다.
→ 한계: 실시간 신작 예측엔 부적합 (한계점 참조).
→ `config.yaml`의 `min_days_since_release`로 조정 가능.

**▶ 왜 리뷰 10개 미만을 제거하나**

숨은 명작 라벨이 긍정률에 의존하는데, 리뷰 3개의 "긍정률 100%"는
통계적으로 무의미하다. 최소 표본을 확보해 라벨을 신뢰 가능하게 만든다.
(숨은 명작은 추가로 30개 이상을 별도 요구)

### [3] 라벨링 — 타겟 2종

**target_hit (흥행 예측)** — `estimated_owners` 구간 하한 ≥ 100,000 → 양성 15.6%
- owners는 "100000 - 200000" 같은 **구간 문자열**이라 하한값을 파싱해 사용
- 10만은 인디 게임 기준 상업적 성공의 통상 경계선

**target_hidden_gem (숨은 명작)** — 다음 3조건 동시 만족 → 양성 19.7%
- 긍정률 ≥ 0.85 (품질 검증)
- owners < 100,000 (아직 묻혀 있음)
- 리뷰 ≥ 30 (긍정률 신뢰)
- hit 기준과 owners 임계값이 같아 **두 라벨은 상호배타**

**▶ 긍정률을 직접 계산한 이유 (데이터 함정)**

데이터에 `pct_pos_total` 컬럼이 있지만, `positive/(positive+negative)`로
직접 계산한 값과 **평균 10.7%p 차이**가 난다. pct_pos_total은 집계 기준이
다른 것으로 보여(전체 리뷰 vs 구매자 리뷰), 일관성을 위해 직접 계산했다.

### [4] 피처 엔지니어링 (83개 생성)

| 그룹 | 피처 | 설명 |
|---|---|---|
| 가격 | price, log_price, is_free | 분포가 오른쪽 꼬리 → 로그 변환. 무료는 별도 플래그 |
| 플랫폼 | windows, mac, linux, n_platforms | 이미 bool 컬럼 |
| 장르/태그/카테고리 | genre_*, tag_*, cat_* + 개수 | 상위 N개 멀티핫 (희소성 방지) |
| 날짜 | year, month, dow, month_sin/cos, is_q4 | 월은 순환 인코딩(12월↔1월 인접) |
| 개발사 | developers_game_count, is_self_published | 이름 대신 규모 통계. 자체퍼블=인디 신호 |

**▶ tags 파싱 주의** — tags는 리스트가 아니라
`{'Survival': 14838, 'Shooter': 12727, ...}` **딕셔너리 문자열**이다.
`ast.literal_eval` 후 dict면 keys만 추출해 멀티핫 처리.

**▶ Leakage 물리 차단 (이 파이프라인의 핵심)**

`positive`, `peak_ccu`, `estimated_owners`, playtime 4종 등
**출시 후에만 알 수 있는 17개 컬럼**은 라벨 생성에만 쓰고 step4에서
DataFrame에서 제거한다. 이후 단계와 팀원 데이터에는 **컬럼이 존재하지
않으므로** 실수로라도 피처에 넣을 수 없다. "주의"가 아니라 "불가능"하게 만든 설계.

### [5] 분할 / 스케일링 / 저장

- **stratify=y** — 불균형(15.6%/19.7%)에서 train/test 양성비를 동일 유지
- **scaler는 train에만 fit** — test 통계 누출(leakage)도 방지. test엔 transform만
- **TF-IDF도 train에만 fit** — short_description을 300차원으로 (DL 텍스트 실험용)
- **데이터 SHA-256 해시 기록** — "어떤 데이터로 학습했나" 추적 (MLOps 재현성)

# 숨은 명작 탐지 모델 — 설명 문서

> 이 문서는 "모델이 뭘 하는 건지, 지표가 뭘 뜻하는지, 이 숫자가 좋은 건지"를 설명하기 위한 문서입니다.

---

## 1. 모델이 하는 일

게임 하나의 정보(가격, 장르, 플랫폼, 출시 시기 등 83개 피처)를 보고
**"이 게임이 숨은 명작일 것 같은 정도"를 0~1 점수로 매기는 것.**

0.9면 "명작일 가능성 높음", 0.1이면 "아닐 것 같음".
7개 모델 전부 이 똑같은 일을 하고, 다만 점수를 매기는 머리 구조가 다릅니다.

---

## 2. 7가지 모델

### 구조

| 모델 | 유형 | 쉬운 설명 |
|---|---|---|
| Logistic Regression | ML | 가장 단순한 일차방정식. "인디면 +3, 비싸면 -1" 식으로 점수 합산. 다른 모델의 기준점(baseline) 역할 |
| Random Forest | ML | 스무고개 나무(트리) 수백 개 만들어서 다수결. 나무 하나가 틀려도 다른 나무들이 보정 |
| XGBoost | ML | Random Forest의 진화형. 나무를 순서대로 만드는데 매번 "이전 나무가 틀린 문제"를 집중 학습. 오답노트 공부법 |
| LightGBM | ML | XGBoost와 같은 원리의 다른 구현체. 속도가 더 빠름 |
| MLP | DL | 인공 뉴런을 여러 층 쌓은 신경망(딥러닝). 이미지·텍스트에선 강하지만, 표 형태 데이터에선 트리 계열한테 지는 경우가 많음 |

**ML vs DL 비교의 의미:** 같은 데이터로 같은 일을 시켰을 때
전통 ML(트리 계열)과 딥러닝(MLP) 중 어느 쪽이 나은지 직접 확인하는 것.

### 불균형 대처법 — 왜 필요한가

숨은 명작이 전체의 9.8%뿐이라서, 모델이 그냥 "전부 명작 아님"이라고 찍어도
90.2% 정답입니다. 게으른 답이 점수가 잘 나오는 구조라, 모델이 진짜로 게을러집니다.
이걸 막는 방법이 둘:

| 방법 | 원리 | 비유 |
|---|---|---|
| **가중치(weight)** | "명작 하나 틀리면 일반 게임 9개 틀린 것만큼 혼낸다" | 채점 기준을 바꿔서 소수를 무시 못 하게 함 |
| **SMOTE(오버샘플링)** | 명작 데이터를 합성으로 뻥튀기해서 학습 시 비율을 맞춤 | 채점이 아니라 교재를 바꾸는 방식 |

### 7개 실험의 조합

| 실험 | 머리 | 불균형 대처 | 유형 |
|---|---|---|---|
| LR_classweight | Logistic Regression | 가중치 | ML |
| RF_classweight | Random Forest | 가중치 | ML |
| XGB_posweight | XGBoost | 가중치 | ML |
| XGB_SMOTE | XGBoost | SMOTE | ML |
| LGBM_unbalance | LightGBM | 가중치 | ML |
| MLP_raw | MLP (딥러닝) | 없음 (그대로) | DL |
| MLP_SMOTE | MLP (딥러닝) | SMOTE | DL |

→ **"어떤 머리가 좋은가(ML vs DL)" × "어떤 불균형 대처가 좋은가(가중치 vs SMOTE)"**
를 동시에 비교하는 설계.

## 3. 지표 — "명작 추천 리스트"

모델이 점수를 매긴 다음, 어느 점수 이상을 "명작"이라고 부를지 선을 긋습니다(임계값).
그러면 결과가 맞춤/틀림으로 갈리고, 지표가 나옵니다.
전부 이 상황으로 연결해서 이해하면 됩니다:
**"모델이 명작 추천 리스트를 만들었다고 치자."**

### Precision (정밀도) — 추천의 신뢰도

추천 리스트에 올린 게임 중 **진짜 명작 비율.**
낮으면 → 유저가 추천 믿고 샀는데 별로인 게임이 많다.

### Recall (재현율) — 명작 발굴률

세상에 있는 진짜 명작 913개 중 **리스트에 담긴 비율.**
낮으면 → 진짜 명작인데 영영 묻히는 게임이 발생.

### 이 둘은 시소

리스트를 길게 만들면(기준을 낮추면) Recall↑ Precision↓ — 다 담으면 발굴은 되지만 잡탕.
짧게 만들면(기준을 높이면) 반대.

### F1 — 시소의 균형 점수

Precision과 Recall 둘 다 적당히 좋아야 높은 점수가 나옴.

### 임계값(threshold) — 시소의 어디에 설지 정하는 손잡이

기본은 0.5인데, **불균형 데이터에선 0.5가 거의 항상 틀린 위치.**

증거: MLP_raw가 0.5 기준으론 F1 0.024 (추천 리스트가 사실상 텅 빔).
임계값을 0.165로 내리니 F1 0.287로 회복.
모델이 멍청한 게 아니라 **선을 잘못 그은 것.**

### PR-AUC — 우리의 1차 평가 지표

임계값을 어디 긋느냐에 따라 점수가 달라지면 모델끼리 공정 비교가 안 됨.
그래서 **가능한 모든 임계값에서의 Precision-Recall 성적을 평균낸 것**이 PR-AUC.
"선을 어디 긋든, 이 모델의 추천 능력 자체는 얼마인가."

### 왜 ROC-AUC가 아닌가

ROC-AUC는 음성(비명작 90%)을 잘 맞히는 것도 점수에 반영돼서,
불균형 데이터에선 과대평가됨.
PR-AUC는 양성(명작 9.8%)에만 집중 — 우리 목적("명작 골라내기")에 정확히 맞음.

---

## 4. 결과 해석

**PR-AUC의 기준선은 0.5가 아닙니다.**

| 지표 | 무작위(찍기) 기준 | 우리 champion | 배수 |
|---|---|---|---|
| ROC-AUC | 항상 0.5 | 아마 0.7~0.8 | ~1.5배 |
| PR-AUC | **양성 비율 = 0.098** | **0.279** | **2.85배** |

무작위로 추천해도 9.8%는 우연히 맞음 → PR-AUC 0.098.
우리 모델은 그 **2.85배** → 찍기 대비 3배 가까이 명작을 잘 골라냄.

PR-AUC
0.10 → 랜덤
0.15 → 별로
0.20 → 쓸만함
0.30 → 꽤 좋음
0.40+ → 매우 좋음

## 5. 실험 결과 전체

| 모델 | 유형 | PR-AUC | F1 (0.5) | F1 (튜닝 후) | 최적 임계값 |
|---|---|---|---|---|---|
| **XGB_SMOTE** | ML | **0.279** | 0.125 | **0.322** | 0.164 |
| XGB_posweight | ML | 0.268 | 0.311 | 0.320 | 0.537 |
| LGBM_unbalance | ML | 0.263 | 0.311 | 0.318 | 0.564 |
| RF_classweight | ML | 0.250 | 0.300 | 0.310 | 0.556 |
| MLP_raw | DL | 0.222 | **0.024** | 0.287 | 0.165 |
| LR_classweight | ML | 0.206 | 0.275 | 0.291 | 0.558 |
| MLP_SMOTE | DL | 0.168 | 0.209 | 0.239 | 0.000 |

### 읽는 법

- **PR-AUC 순서가 모델의 진짜 실력 순서.** 임계값에 흔들리지 않는 종합 점수.
- **F1(0.5)**: 기본 임계값 0.5에서의 균형 점수. XGB_SMOTE와 MLP_raw가 낮은 건
  0.5가 안 맞는 거지 모델이 멍청한 게 아님.
- **F1(튜닝 후)**: 최적 임계값에서의 점수. 이게 실제 서비스에 쓸 때의 성능.

---

## 6. 핵심 발견 3가지

### ① ML이 DL을 이겼다

ML 최고(XGB_SMOTE 0.279) vs DL 최고(MLP_raw 0.222) — 전 구간에서 ML 우세.
정형 데이터 + 4.6만 건 규모에서는 트리 기반 부스팅이 신경망보다 유리함을 실증.

### ② SMOTE는 양날의 검

XGB+SMOTE는 PR-AUC 1위(0.279)지만 기본 F1은 0.125로 최하위권.
SMOTE가 학습 데이터를 부풀리면서 확률 분포를 왜곡해,
기본 임계값이 안 맞게 됨. 임계값 튜닝과 같이 써야 함.

### ③ 임계값이 모델만큼 중요하다

MLP_raw — F1 0.024(추천 리스트 텅 빔) → 임계값 조정 → F1 0.287(회복).
불균형 데이터에서 0.5는 "신성한 숫자"가 아니라 그냥 기본값일 뿐.
